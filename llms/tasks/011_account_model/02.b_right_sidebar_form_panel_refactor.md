## Design Direction

- Use the same page rhythm already established in `EntitiesLive` and `AccountsLive`:
  - page header
  - top explanatory callout
  - primary list content
  - lower guidance section
- The right sidebar should feel like a reusable application pattern, not a one-off modal.
- Prefer a slide-over/right-docked panel over a centered modal.
- Keep accessibility in mind:
  - clear close button
  - readable heading/title
  - sensible focus order
  - overlay/background separation

## Technical Notes

### Scope boundary
- The reusable component should own the visual shell only.
- The concrete form content should remain specific to each feature and be passed via slots.
- Do **not** move entity/account domain logic into the shared sidebar component.

### Suggested state shape
Each LiveView may introduce something like:
- `:form_open?`
- existing `:editing_entity` / `:editing_account`

Typical behavior:
- `new_*` -> open sidebar + reset editing target
- `edit_*` -> open sidebar + load editing target
- successful save -> close sidebar + reset form
- explicit close/cancel -> close sidebar + reset form state

### Stable DOM IDs
Suggested IDs:
- `#right-sidebar-panel`
- `#right-sidebar-overlay`
- `#close-sidebar-btn`
- keep feature form IDs stable (`#entity-form`, `#account-form`)

### Constraints
- Do not redesign the form fields themselves beyond what is needed for placement in the sidebar
- Do not alter business logic or validations
- Do not remove current guidance/info sections from the page body
- Do not introduce LiveComponents unless clearly justified; prefer function components + LiveView state

### Assumptions Made
| Assumption | Rationale |
|------------|-----------|
| A single generic sidebar shell can be reused across page-level LiveViews | Both pages only need a visual shell; the concrete forms remain feature-specific |
| Generic DOM IDs like `#right-sidebar-panel` are acceptable | Only one sidebar is rendered per page at a time, so selectors remain stable |

### Decisions Made
| Decision | Alternatives Considered | Rationale |
|----------|------------------------|-----------|
| Keep the sidebar shell as a function component, not a LiveComponent | LiveComponent, inline duplicate markup | The shell has no state of its own and only needs slots + parent LiveView events |
| Close and reset form state on successful save and explicit close | Keep stale form state mounted | Resetting prevents editing residue when switching flows |
